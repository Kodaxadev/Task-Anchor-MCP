# Drift vs. Valid Scope

## Drift Patterns (Park Immediately)
| User Signal | Translation | Why It's Drift |
|-------------|-------------|----------------|
| "Actually, let's..." | Abandoning current direction mid-stream | Creates orphan code, cognitive residue |
| "While I'm at it..." | Scope creep disguised as efficiency | ADHD hyperfocus trap—3 hours disappear |
| "What if we used [New Tech]..." | Novelty seeking mid-task | Dopamine chase, not delivery |
| "Can you explain how [unrelated thing] works?" | Research tangent | Learning mode ≠ Delivery mode |
| "This reminds me of [other project]..." | Context switching | Brain is trying to escape stuck-ness |

## Valid Scope Expansion (Proceed)
| Signal | Validation Check |
|--------|------------------|
| Bug discovered | Is it IN the files listed in SCOPE FILES? |
| Refactoring needed | Is it required to make the current task work? |
| Architecture question | Does it unblock the current EXIT CONDITION? |
| Explicit completion | User explicitly stated current task is done |

## Edge Cases
- **"I need to google this"** → Park. Research is a different task.
- **"This is taking too long, let's start over differently"** → Validate if stuck >30 mins. If yes, mark current failed, park new approach as new task.
- **Emergency production fix** → Valid interrupt, but must explicitly mark current task "Suspended" with resume notes.
